## 🔥 <font id=前言>前言</font>

> 本协议约束 Codex 如何读取、写入、去重、复盘和归档本 Vault。

## 一、任务前读取 <a href="#前言" style="font-size:17px; color:green;"><b>🔼</b></a> <a href="#🔚" style="font-size:17px; color:green;"><b>🔽</b></a>

1. 扫描 Vault 目录与 Markdown 文件清单。
2. 读取 `00-记忆入口.md`。
3. 读取当前任务命中的长期偏好、工作流决策和项目记忆。
4. 只把命中正文用于当前任务；不全量灌入无关文件。

## 二、记忆质量门槛

- 必须满足至少一项：跨任务长期规划、用户稳定偏好、已确认工作流决策、可复用项目约束。
- 必须有来源，并能区分“用户明确确认”、“从文件系统实测”和“待确认候选”。
- 不得保存密码、Token、私钥、Cookie、一次性日志和可随时重新查询的短期事实。

## 三、分类与去重

- 按“用户偏好→长期规划→工作流决策→项目记忆”顺序选择最具体的主位置。
- 先搜索同义词、仓库路径和关键约束；命中时更新原条目，不叠加重复记忆。
- 冲突时保留旧决策，标记 `superseded`并链接新决策；尚无法判定时进入收集箱，不擅自覆盖。

## 四、每日凌晨复盘

1. 读取 `90-系统/同步状态.md`，以 `last_successful_review_at` 为起点。
2. 检查该游标之后完成的 Codex 任务；优先使用 Codex 任务列表与任务摘要，并用 `~/.codex/memories/MEMORY.md` 和相关 rollout summary 交叉校验。
3. 仅提取达到质量门槛的新增或更新项，完成去重和冲突处理。
4. 写入主记忆并更新 `00-记忆入口.md`，然后追加当日复盘笔记。
5. 全部验证通过后才更新 `last_successful_review_at`；失败时保留原游标并记录阻断原因。
6. 无新增时仍记录检查范围和“无新增”，再更新成功游标。

## 五、验证

- 新文件必须带有 Obsidian Properties 所需的 YAML frontmatter。
- 索引中的 Wiki 链接必须指向实际存在的 Markdown 文件或标题。
- 同步状态与当日复盘的时间、新增数和更新数必须一致。
